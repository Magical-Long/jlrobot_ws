## 笔记 1：势场法避障实现的局限与改进方向

本节结合 `src/robot_core/src/obstacle_avoidance_planner.cc` 的当前实现，说明这版机械臂势场法避障规划器的主要缺点、容易出问题的场景，以及后续可以怎样改进。

目标不是否定当前实现，而是帮助后续迭代时更清楚地知道：

- 这版代码适合做什么
- 不适合做什么
- 如果实验效果不好，优先应该从哪里改

### 1.1 当前实现的基本思路

当前规划器每一步都在关节空间构造一个总步进：

`joint_step = attractive_step + repulsive_step + joint_centering_step`

其中：

- `attractive_step`：把末端往目标位姿拉过去
- `repulsive_step`：把若干链路 frame 从障碍物旁边推开
- `joint_centering_step`：把关节往上下限中间拉，减少贴边

然后再做：

- 单步范数限幅
- 速度限幅
- `pinocchio::integrate()`
- 关节上下限裁剪

对应代码主要在：

- `planToPose()`
- `computeAttractiveJointStep()`
- `computeRepulsiveJointStep()`
- `computeJointCenteringStep()`

### 1.2 这版实现的优点

先说优点，这样后面看缺点时不会失去上下文。

- 结构简单，容易读懂，便于教学和调试
- 不需要复杂优化器，运行代价较低
- 直接复用 Pinocchio 的正运动学和 Jacobian，工程实现比较干净
- 对单个或少量球形障碍物、较空旷工作空间，通常能得到可用结果
- 参数较直观，例如 `repulsive_gain`、`influence_distance`、`max_joint_step_norm`

所以，这版代码很适合作为：

- 势场法入门示例
- 机械臂避障原型验证
- 后续更复杂规划器的基线版本

### 1.3 主要缺点

#### 1.3.1 容易陷入局部极小值

这是势场法最经典的问题。

当前实现中，末端吸引项、障碍物排斥项和关节居中项是直接相加的。某些位置上，这几个方向可能互相抵消，导致：

- 机械臂离目标还很远，但更新量已经很小
- 障碍物左右两边排斥力对冲
- 末端想往前，链路又被障碍物推回来

代码里已经有一个停滞判定：

`if ((q - joint_waypoints.back()).norm() < 1.0e-6) { break; }`

这说明当前实现也默认承认“有可能走不动”。但这里的处理只是提前退出，并没有主动跳出局部极小值。

典型表现：

- 路径在障碍物前面停住
- 末端在目标附近来回抖动但进不去
- 明明存在可行路径，但算法返回失败

#### 1.3.2 多个排斥项直接叠加，可能互相打架

当前 `computeRepulsiveJointStep()` 的逻辑是：

1. 遍历所有 `collision_frames`
2. 遍历所有球形障碍物
3. 对每一对 `frame-obstacle` 计算一个排斥方向
4. 直接累加到 `repulsive_step`

这意味着所有局部建议最后都变成一个“合力”。

问题在于：

- 某个 frame 想往左躲，另一个 frame 想往右躲
- 上部链路想抬高，下部链路想压低
- 某个障碍物的回避动作会让另一个障碍物更近

因此最终的 `repulsive_step` 很可能不是任何单个障碍物的最优回避方向，而只是一个折中结果。

这会带来两个后果：

- 排斥效果变弱
- 在狭窄空间里更容易失败

#### 1.3.3 排斥项只看位置，不看姿态和真实几何体

当前排斥项使用的是：

- 某个 `frame` 的世界坐标位置
- 球形障碍物中心
- 二者之间的净距离

并且 Jacobian 只取了线速度部分：

`const Eigen::MatrixXd linear_jacobian = frame_jacobian.topRows(3);`

这说明当前避障逻辑只关注“点离障碍物远不远”，没有考虑：

- link 本身的长度和体积
- link 的朝向是否会造成扫碰
- 连杆中间部分是否已经穿过障碍物

换句话说，这版实现更像：

- 用若干采样点近似整条机械臂

而不是：

- 对整条机械臂几何体做严格碰撞检测

因此会存在漏检风险，尤其在这些场景更明显：

- 长连杆从障碍物旁边掠过
- frame 点没有碰到，但 link 中段已经碰到
- 末端安全，但前臂已经进入障碍物附近

#### 1.3.4 障碍物模型过于理想化

当前实现只支持 `SphericalObstacle`，即球形障碍物。

这当然很方便，因为：

- 距离好算
- 梯度好写
- 参数少

但真实环境中的障碍物常常不是球体，例如：

- 桌面是平面
- 机柜是盒体
- 墙体是大平面或多边形
- 工装夹具是非凸结构

如果全部强行近似成球，会出现两个问题：

- 近似过粗，导致绕得太保守
- 近似不准，导致本应碰撞的地方被漏掉

#### 1.3.5 缺少主任务与次任务的优先级机制

现在的吸引项、排斥项、关节居中项都在同一个层面直接相加，这意味着：

- 没有严格的主从关系
- 没有“先满足末端任务，再利用冗余度避障”的结构

对于 7 自由度机械臂，更合理的思路通常是：

- 主任务：末端跟踪目标位姿
- 次任务：在主任务零空间中尽量避障
- 再次任务：远离关节极限

而当前实现不是零空间分层，而是简单加权求和。所以常见问题是：

- 避障太强时，末端任务被拖偏
- 吸引太强时，避障来不及生效
- 参数只能靠手工反复调

#### 1.3.6 对参数非常敏感

当前配置文件里有很多强相关参数：

- `attractive_position_gain`
- `attractive_orientation_gain`
- `repulsive_gain`
- `influence_distance`
- `joint_centering_gain`
- `jacobian_damping`
- `sample_period`
- `max_joint_step_norm`

这些参数之间不是独立的。

例如：

- `repulsive_gain` 太小，障碍物几乎没效果
- `repulsive_gain` 太大，机械臂可能被“弹飞”
- `influence_distance` 太短，发现障碍物太晚
- `sample_period` 太大，数值积分会显得粗糙
- `max_joint_step_norm` 太小，算法会走得很慢

这意味着当前实现对不同场景的泛化能力有限，往往需要针对具体任务重复调参。

#### 1.3.7 没有真正的碰撞检测，只是“距离驱动”

当前代码不是严格回答“碰撞了没有”，而是在回答：

- 某些 frame 离障碍物近不近

这两者不是一回事。

严格碰撞检测通常需要：

- 机械臂几何体模型
- 障碍物几何体模型
- 最近距离求解
- 碰撞判定或 signed distance

而当前代码只是根据人工选定的 `collision_frames` 做近似。这在演示时可以接受，但在更严肃的场景中会带来问题：

- 你以为在避障，其实只是“部分点在避障”
- 你以为没碰撞，其实只是不在采样点上碰撞

#### 1.3.8 收敛失败时缺少更细致的诊断信息

当前 `planToPose()` 返回失败时，上层只知道：

- 没在 `max_steps` 内到达目标
- 或中途某个计算失败

但并不知道到底是哪一种失败：

- 是被局部极小值困住
- 是吸引项和排斥项冲突
- 是姿态任务太强
- 是速度限幅过严
- 是初值本来就很差

这会让调参和排错变得比较低效。

### 1.4 典型容易失败的场景

结合当前实现，以下场景最容易出问题。

#### 1.4.1 狭窄通道

比如两个障碍物之间刚好能通过，但空间很窄。

在这种情况下：

- 左边障碍物往右推
- 右边障碍物往左推

结果常常是：

- 排斥项互相抵消
- 机械臂停在通道口

#### 1.4.2 需要“先远离目标，再绕过去”的路径

势场法天然倾向于局部贪心。  
如果某个可行路径要求机械臂先后退、再绕开、最后再前进，那么当前吸引项可能一直拉着末端朝目标方向冲，导致绕不过去。

#### 1.4.3 长连杆扫碰

如果只检测 `iiwa_link_1` 到 `iiwa_link_ee` 这些 frame 点，而不检测 link 的连续几何体，那么：

- frame 点可能都安全
- 但连杆中间某段已经擦到障碍物

#### 1.4.4 目标姿态要求较强

当 `hold_orientation = true` 时，末端不但要到位置，还要保持姿态。  
这会进一步压缩冗余度，使得：

- 可用于避障的自由度更少
- 局部冲突更严重

### 1.5 可以改进的方向

下面按“从容易实现到更系统化”的顺序给出一些建议。

#### 1.5.1 给排斥项加权，而不是简单全加

可以优先增强最危险的贡献项，例如：

- 只保留最近的几个 `frame-obstacle` 对
- 按净距离倒数平方做归一化权重
- 对已经进入安全边界的项给更高权重

这样做的好处是减少“大家都说话，谁都不算数”的情况。

#### 1.5.2 引入更细的碰撞采样点

现在是按 `collision_frames` 做检测。可以进一步改进为：

- 每个 link 上增加多个采样点
- 沿连杆均匀插值几个虚拟点
- 对关键长连杆加密采样

这样仍然保持实现简单，但能明显减少“中段漏检”。

#### 1.5.3 改用真实几何距离或 signed distance

更进一步的方案是引入几何碰撞库，例如：

- FCL
- hpp-fcl

目标是从“frame 点到球心距离”升级为：

- link 几何体到障碍物几何体的最近距离
- 或 signed distance / penetration depth

这样避障项会更接近真实碰撞约束。

#### 1.5.4 使用零空间投影做任务分层

对于 7 轴冗余机械臂，一个常见改进是：

`dq = dq_task + N(q) * dq_avoid`

其中：

- `dq_task` 先满足末端主任务
- `N(q)` 是主任务 Jacobian 的零空间投影
- `dq_avoid` 在不破坏主任务的一阶近似前提下尽量避障

这样会比“所有项直接相加”更有结构，也更符合冗余机械臂控制的常见做法。

#### 1.5.5 从势场法升级到约束优化

如果希望更稳定地处理复杂障碍物和多任务冲突，可以把问题写成 QP 或 NLP：

- 优化变量：`dq`
- 目标函数：跟踪末端目标、保持姿态平滑、远离关节极限
- 约束：碰撞距离下界、速度上限、位置上限

这样做的优点是：

- 每一步的目标和约束更明确
- 更容易表达“不能碰撞”这种硬约束

缺点是：

- 实现复杂度更高
- 需要引入优化器

#### 1.5.6 增强失败诊断与日志输出

建议在 `planToPose()` 里增加调试信息，例如：

- 每一步的位置误差和姿态误差
- 最近障碍物距离
- 吸引项、排斥项、关节居中项各自的范数
- 因速度限幅被裁剪了多少
- 是因为达到 `max_steps` 失败，还是因为步长停滞失败

这样调参时会快很多。

#### 1.5.7 针对局部极小值增加恢复策略

例如：

- 停滞时随机扰动一个小步
- 暂时减弱姿态保持
- 暂时提高最近障碍物排斥权重
- 切换到另一组初值重新规划

这类方法不一定最优，但在工程上往往很实用。

### 1.6 一个比较务实的升级路线

如果希望在不大改架构的前提下逐步提升效果，可以按下面顺序做。

#### 第一阶段：低成本增强

- 增加日志和失败原因统计
- 对最近障碍物和最近 frame 提高权重
- 给关键 link 增加更多采样点
- 区分“位置任务”和“姿态任务”的权重调节策略

#### 第二阶段：中等改动

- 使用零空间投影做主从任务分层
- 引入更合理的排斥势函数
- 加入停滞恢复策略

#### 第三阶段：系统升级

- 引入真实几何碰撞距离
- 改为 QP / 约束优化框架
- 支持盒体、胶囊体、平面等更多障碍物类型

### 1.7 总结

当前这版势场法避障规划器是一个不错的原型版本，它的优势在于：

- 简单
- 清晰
- 易于上手
- 能跑起来

但它的核心局限也很明确：

- 容易局部极小值
- 多个排斥项会互相冲突
- 只做点级近似，不做真实碰撞检测
- 参数敏感，泛化性有限
- 缺少主从任务分层

所以如果目标是：

- 学习势场法
- 快速做实验
- 搭建原型

这版实现是合适的。

如果目标是：

- 稳定通过复杂障碍环境
- 在狭窄空间中可靠规划
- 作为更正式的机械臂避障模块上线

那么后续就需要继续往零空间控制、真实距离场、碰撞检测或约束优化方向演进。
